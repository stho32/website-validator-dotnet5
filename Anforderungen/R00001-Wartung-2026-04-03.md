**Identifizierte Kern-Features (aus README und Code):**

1. **URL-Crawling** — Website ab einer Start-URL crawlen, alle internen Links verfolgen
2. **Link-Extraktion** — HTML-Links (a href) und Bild-URLs (img src) extrahieren
3. **Relative-zu-Absolute URL-Konvertierung** — Relative URLs in absolute umwandeln
4. **HTTP-Status-Pruefung** — HTTP-Statuscode jeder Seite erfassen
5. **JSON-Output** — Ergebnisse als JSON-Datei oder auf Konsole ausgeben
6. **Human-Readable Output** — Menschenlesbare Konsolenausgabe
7. **SSL-Ignorierung** — Option zum Ignorieren von SSL-Zertifikatsproblemen
8. **Crawl-Limit** — Begrenzung der maximal zu crawlenden Seiten
9. **Zusaetzliche Entry Points** — Weitere Start-URLs aus Datei laden

**Geplante, nicht umgesetzte Features (aus README):**
- Externe URLs nur pruefen, nicht weiter crawlen
- CSS/JS-Ressourcen crawlen
- Konfigurierbares Validierungssystem
- Sitemap-Abgleich
- robots.txt-Abgleich
- Muting von bekannten Validierungsmeldungen

## Phase 3: Relevanz-Bewertung

### KI-Entwicklung & Disruption

KI-basierte Tools im Website-Validierungsbereich sind primaer als Erweiterung bestehender SEO-Suiten praesent (z.B. KI-gestuetzte Content-Analyse in Screaming Frog, Sitebulb). Der Kern-Use-Case "Link-Checking/Crawling" ist ein klassisches, deterministisches Problem, das nicht durch KI ersetzt wird. KI koennte den Validator allerdings aufwerten: intelligente Fehlerklassifizierung, Content-Qualitaetsanalyse, automatische Fix-Vorschlaege.

### Markt & Alternativen

| Tool | Typ | Staerke |
|------|-----|---------|
| **Screaming Frog SEO Spider** | Kommerziell | Marktfuehrer, extrem umfangreich |
| **linkchecker** (Python) | Open-Source | Ausgereift, aktiv gepflegt, CLI |
| **broken-link-checker** (Node.js) | Open-Source | Einfach, populaer |
| **Lighthouse** (Google) | Open-Source | Browser-basiert, Performance + Accessibility |
| **muffet** (Go) | Open-Source | Sehr schnell, CLI |
| **lychee** (Rust) | Open-Source | Extrem schnell, modern, aktiv gepflegt |
| **Zed Attack Proxy (ZAP)** | Open-Source | Security-fokussiert |

**Positionierung:** Ein kleines .NET 5 CLI-Tool ohne besondere Alleinstellungsmerkmale hat im Vergleich zu ausgereiften Alternativen wie `lychee` (Rust, extrem schnell) oder `linkchecker` (Python, feature-reich) kaum Differenzierung. Die JSON-Output-Funktion ist nuetzlich, wird aber von allen modernen Alternativen ebenfalls angeboten.

### Nutzungsmuster & Aktivitaet

- **Letzter Commit:** 2021-08-30 (vor ~4,5 Jahren)
- **Commit-Frequenz:** 63 Commits in ~10 Monaten (Nov 2020 — Aug 2021), danach Stillstand
- **Contributors:** 4 (hauptsaechlich stho32)
- **Releases:** 9 Releases (v1.0.1 bis v1.0.9)
- **Upstream-Dependencies:** .NET 5 ist EOL, Newtonsoft.Json hat Sicherheitsluecke

### Relevanz-Ampel

| Dimension | Bewertung | Einschaetzung |
|-----------|-----------|---------------|
| KI-Entwicklung | 🟢 | Link-Checking ist deterministisch, KI ersetzt den Use-Case nicht |
| Markt & Alternativen | 🔴 | Zahlreiche ausgereiftere, schnellere Open-Source-Alternativen verfuegbar (lychee, linkchecker, muffet) |
| Nutzung & Aktivitaet | 🔴 | Seit 4,5 Jahren inaktiv, EOL-Framework, keine Community |
| **Gesamt** | 🔴 | **Abloesen oder grundlegend modernisieren** |

**Empfehlung:** Das Projekt hat seinen Zweck als Lern-/Hobby-Projekt erfuellt. Fuer produktiven Einsatz empfiehlt sich die Migration auf ein etabliertes Tool wie `lychee` (Rust) oder `linkchecker` (Python). Falls Weiterentwicklung gewuenscht: Komplette Modernisierung auf .NET 8/9 mit Differenzierung durch spezifische Features (z.B. Validierungsregeln, Sitemap-Abgleich, Reporting).

### Vorschlag 2: Newtonsoft.Json durch System.Text.Json ersetzen

**Kategorie:** Sicherheit
**Prioritaet:** Hoch
**Aufwand:** Klein

**Ist-Zustand:** Newtonsoft.Json 12.0.3 hat eine bekannte Sicherheitsluecke (GHSA-5crp-9r3c-p9vr). Ausserdem ist System.Text.Json seit .NET Core 3.0 im Framework enthalten und performanter.

**Soll-Zustand:** Verwendung von `System.Text.Json` aus dem .NET SDK.

**Umsetzung:**
1. In `JsonConsoleOutputHelper.cs` und `JsonFileOutputHelper.cs`: `JsonConvert.SerializeObject` durch `System.Text.Json.JsonSerializer.Serialize` ersetzen
2. `Newtonsoft.Json` NuGet-Paket entfernen aus `WebsiteValidator.BL.csproj`
3. Serialization-Optionen anpassen (z.B. `JsonSerializerOptions { WriteIndented = true }`)

### Vorschlag 3: Async/Await korrekt implementieren

**Kategorie:** Anwendung & Code
**Prioritaet:** Hoch
**Aufwand:** Mittel

**Ist-Zustand:** In `Crawler.cs:55` und `WebpageExtensionMethods.cs:10` wird `.Result` auf Tasks aufgerufen, was den Thread blockiert und in bestimmten Kontexten zu Deadlocks fuehren kann. Die `Main`-Methode ist synchron.

**Soll-Zustand:** Durchgaengige async/await-Kette von `Main` bis `DownloadAWebpage.Download`.

**Umsetzung:**
1. `Program.Main` auf `async Task<int>` aendern
2. `Crawler.CrawlEverything` auf `async Task<IUrlInformation[]>` aendern
3. `WebpageExtensionMethods.ExtractUrls` auf `async Task<string[]>` aendern und `await` statt `.Result` verwenden
4. Alle Aufrufer entsprechend anpassen

### Vorschlag 4: URL-Converter fuer http:// und Sonderfaelle korrigieren

**Kategorie:** Anwendung & Code
**Prioritaet:** Hoch
**Aufwand:** Klein

**Ist-Zustand:** `UrlToAbsolutUrlConverter.ToAbsoluteUrl()` (Zeile 16) prueft nur auf `https://`. URLs die mit `http://` beginnen, werden faelschlicherweise als relative URLs behandelt und erhalten den Base-URL als Prefix. Ausserdem werden `mailto:`, `tel:`, `javascript:` und Fragment-Links (`#`) nicht behandelt.

**Soll-Zustand:** Korrekte Behandlung aller URL-Schemata.

**Umsetzung:**
1. Pruefung auf `http://` und `https://` (oder allgemein: `Uri.IsWellFormedUriString` mit `UriKind.Absolute`)
2. `mailto:`, `tel:`, `javascript:`, `data:` URLs ignorieren/filtern
3. Fragment-only Links (`#section`) korrekt behandeln
4. Bestehende Tests erweitern, neue Testfaelle hinzufuegen

### Vorschlag 8: HTTP-Timeouts und Rate-Limiting implementieren

**Kategorie:** Sicherheit
**Prioritaet:** Mittel
**Aufwand:** Klein

**Ist-Zustand:** `DownloadAWebpage.Download()` hat keinen Timeout konfiguriert. Der Crawler macht Requests so schnell wie moeglich ohne Pause zwischen Requests. Dies kann bei langsamen Servern zum Haengenbleiben fuehren und den Ziel-Server ueberlasten.

**Soll-Zustand:** Konfigurierbarer Timeout und optionales Rate-Limiting.

**Umsetzung:**
1. `HttpClient.Timeout` setzen (z.B. 30 Sekunden)
2. Optional: `--timeout` CLI-Option hinzufuegen
3. Optional: `--delay` CLI-Option fuer Pause zwischen Requests
4. User-Agent Header setzen (hoefliches Crawling)

### Strategische Einordnung

**Gesamtbewertung: 🔴 Grundlegende Modernisierung oder Abloesung erforderlich**

Das Projekt ist ein funktionsfaehiger Website-Crawler mit solider Grundarchitektur (Interface-Trennung, Factory Pattern), aber:

- **Technisch veraltet:** .NET 5 EOL, veraltete Dependencies mit Sicherheitsluecken, Tests nicht ausfuehrbar
- **Seit 4,5 Jahren inaktiv:** Kein Commit seit August 2021
- **Starke Konkurrenz:** Ausgereiftere Open-Source-Tools wie `lychee` (Rust), `linkchecker` (Python) oder `muffet` (Go) bieten mehr Features, bessere Performance und aktive Community

**Handlungsoptionen:**

1. **Weiterentwickeln** — Lohnt sich nur mit klarem Differenzierungsmerkmal (z.B. .NET-Oekosystem-Integration, spezifische Validierungsregeln, PowerShell-Integration). Erfordert mindestens Vorschlaege 1-8.

2. **In Wartungsmodus setzen** — Minimale Aktualisierung (.NET 8, Security-Fixes), dann archivieren. Fuer Nutzer die bereits darauf setzen.

3. **Abloesung empfehlen** — Fuer neue Projekte auf `lychee` oder `linkchecker` verweisen. Repository archivieren.

**Empfehlung:** Option 2 oder 3, je nach aktueller Nutzung. Falls das Tool aktiv genutzt wird, zuerst die Sicherheits-Vorschlaege (1, 2, 6) umsetzen und dann entscheiden.
